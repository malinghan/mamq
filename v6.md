# v6.0 — 可靠投递

## 1. 版本目标

消费失败不丢消息，支持自动重试和死信队列。消息支持 TTL 过期自动跳过。

**客户端改动**：`MABroker` 增加重试逻辑和 DLQ 转发。
**服务端改动**：`recv` 时检查 TTL，过期消息自动跳过。

---

## 2. 核心功能说明

### 三个新特性

**重试（Retry）**：push 模式下，`listener.onMessage()` 抛异常时，`MABroker` 按退避策略重试（1s、2s、4s），最多 3 次。

**死信队列（DLQ）**：超过最大重试次数的消息，转发到 `{topic}.DLQ` Topic，等待人工处理或告警。

**TTL（Time To Live）**：消息 headers 携带 `X-ttl`（毫秒）和 `X-send-time`（发送时间戳），`recv` 时检查是否过期，过期则自动跳过并推进 offset。

### 重试流程

```
MABroker 轮询线程
      │
      ▼
  consumer.recv(topic)
      │
      ▼
  msg != null ?
  ┌───┴───┐
  │ Yes   │ No → sleep 100ms
  ▼
  listener.onMessage(msg)
      │
      ▼
  抛出异常？
  ┌───┴───┐
  │ Yes   │ No → consumer.ack(msg) → 完成
  ▼
  retryCount < 3 ?
  ┌───┴───┐
  │ Yes   │ No
  ▼       ▼
退避重试  转发到 {topic}.DLQ
(1s/2s/4s) consumer.ack(原消息)
  │
  ▼
继续轮询
```

### TTL 检查流程

```
MessageQueue.recv(cid)
      │
      ▼
  store.read(offset) → msg
      │
      ▼
  msg.headers["X-ttl"] 存在？
  ┌───┴───┐
  │ Yes   │ No → 正常返回 msg
  ▼
  当前时间 > sendTime + ttl ?
  ┌───┴───┐
  │ Yes   │ No → 正常返回 msg
  ▼
  自动 ack 跳过，offset++，读取下一条
  （循环直到找到未过期消息或无消息）
```

## 4. 代码实现说明

### MessageQueue.java（TTL 检查）

```java
public Message recv(String consumerId, String groupId) {
    if (groupId == null) {
        return recvIndependentWithTTL(consumerId);
    }
    return recvGroupWithTTL(groupId);
}

private Message recvIndependentWithTTL(String consumerId) {
    Subscription sub = subscriptions.get(consumerId);
    if (sub == null) return null;

    while (sub.getOffset() < store.size()) {
        int offset = sub.getOffset();
        Message msg = store.read(offset);
        if (msg == null) return null;

        // 检查 TTL
        if (isExpired(msg)) {
            sub.setOffset(offset + 1);  // 自动跳过过期消息
            persistSubscriptions();
            continue;
        }

        msg.setOffset(offset);
        return msg;
    }
    return null;
}

private synchronized Message recvGroupWithTTL(String groupId) {
    Subscription groupSub = groupSubscriptions.get(groupId);
    if (groupSub == null) return null;

    while (groupSub.getOffset() < store.size()) {
        int offset = groupSub.getOffset();
        Message msg = store.read(offset);
        if (msg == null) return null;

        if (isExpired(msg)) {
            groupSub.setOffset(offset + 1);  // 自动跳过
            continue;
        }

        groupSub.setOffset(offset + 1);  // 预占
        msg.setOffset(offset);
        return msg;
    }
    return null;
}

private boolean isExpired(Message msg) {
    if (msg.getHeaders() == null) return false;
    String ttlStr = msg.getHeaders().get("X-ttl");
    String sendTimeStr = msg.getHeaders().get("X-send-time");
    if (ttlStr == null || sendTimeStr == null) return false;

    long ttl = Long.parseLong(ttlStr);
    long sendTime = Long.parseLong(sendTimeStr);
    long now = System.currentTimeMillis();
    return now > sendTime + ttl;
}
```

### MABroker.java（重试 + DLQ）

```java
public class MABroker {
    private static final int MAX_RETRY = 3;
    private static final long[] RETRY_DELAYS = {1000, 2000, 4000}; // 1s, 2s, 4s

    // ... 其他代码不变

    @SuppressWarnings("unchecked")
    private <T> void pollOnce(MAConsumer<T> consumer) throws InterruptedException {
        Message<T> msg = consumer.recv();
        if (msg == null) {
            Thread.sleep(100);
            return;
        }

        if (consumer.listener != null) {
            boolean success = processWithRetry(consumer, msg);
            if (success) {
                consumer.ack(msg.getOffset());
            } else {
                // 超过最大重试次数，转发到 DLQ
                sendToDLQ(consumer, msg);
                consumer.ack(msg.getOffset());  // ACK 原消息，避免重复消费
            }
        }
    }

    private <T> boolean processWithRetry(MAConsumer<T> consumer, Message<T> msg) {
        for (int retry = 0; retry < MAX_RETRY; retry++) {
            try {
                consumer.listener.onMessage(msg);
                return true;  // 成功
            } catch (Exception e) {
                System.err.println("消费失败，重试 " + (retry + 1) + "/" + MAX_RETRY
                    + ": " + e.getMessage());
                if (retry < MAX_RETRY - 1) {
                    try {
                        Thread.sleep(RETRY_DELAYS[retry]);
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        return false;
                    }
                }
            }
        }
        return false;  // 重试耗尽
    }

    private <T> void sendToDLQ(MAConsumer<T> consumer, Message<T> msg) {
        try {
            String dlqTopic = consumer.topic + ".DLQ";
            MAProducer producer = createProducer();
            // 添加失败原因到 headers
            if (msg.getHeaders() == null) {
                msg.setHeaders(new HashMap<>());
            }
            msg.getHeaders().put("X-dlq-reason", "max-retry-exceeded");
            msg.getHeaders().put("X-dlq-time", String.valueOf(System.currentTimeMillis()));
            producer.send(dlqTopic, msg);
            System.err.println("消息转入死信队列: " + dlqTopic + ", offset=" + msg.getOffset());
        } catch (Exception e) {
            System.err.println("转发 DLQ 失败: " + e.getMessage());
        }
    }
}
```

### 场景二：TTL 过期验证

```bash
# Step 1: 发一条 TTL=2000ms（2秒）的消息
SEND_TIME=$(date +%s%3N)  # 毫秒时间戳
curl -s -X POST "http://localhost:8765/mamq/send?t=test.v6" \
  -H "Content-Type: application/json" \
  -d "{\"id\":1,\"body\":\"will-expire\",\"headers\":{\"X-ttl\":\"2000\",\"X-send-time\":\"$SEND_TIME\"},\"offset\":null}"

# Step 2: 立即消费（未过期）
curl -s "http://localhost:8765/mamq/sub?t=test.v6&cid=c1"
curl -s "http://localhost:8765/mamq/recv?t=test.v6&cid=c1"
# 期望: 拿到 body="will-expire" 的消息

# Step 3: 不 ACK，等待 3 秒后再消费
sleep 3
curl -s "http://localhost:8765/mamq/recv?t=test.v6&cid=c1"
# 期望: 仍然拿到同一条消息（因为未 ACK，offset 未推进）

# Step 4: ACK 后再发一条 TTL=2000ms 的消息
curl -s "http://localhost:8765/mamq/ack?t=test.v6&cid=c1&offset=0"
SEND_TIME=$(date +%s%3N)
curl -s -X POST "http://localhost:8765/mamq/send?t=test.v6" \
  -H "Content-Type: application/json" \
  -d "{\"id\":2,\"body\":\"will-expire-2\",\"headers\":{\"X-ttl\":\"2000\",\"X-send-time\":\"$SEND_TIME\"},\"offset\":null}"

# Step 5: 等待 3 秒后消费（已过期）
sleep 3
curl -s "http://localhost:8765/mamq/recv?t=test.v6&cid=c1"
# 期望: {"code":1,"data":null}  ← 消息已过期被自动跳过

# Step 6: 查看统计，验证 offset 已自动推进
curl -s "http://localhost:8765/mamq/stat?t=test.v6&cid=c1"
# 期望: {"code":1,"data":{"total":2,"position":2}}  ← position=2 说明过期消息被跳过
```

## 6. 版本总结

### 学到了什么

v6.0 实现了消息系统的可靠投递三要素：

**重试**：临时故障（网络抖动、依赖服务短暂不可用）不丢消息，指数退避避免雪崩。

**死信队列**：永久故障（消息格式错误、业务逻辑 bug）不阻塞队列，转入 DLQ 等待人工介入。

**TTL**：过期消息自动跳过，避免消费无效数据（如限时优惠券、实时推荐）。

```
消费流程

recv → TTL 检查 → 过期？→ 自动跳过，读下一条
                    ↓ 未过期
                listener.onMessage()
                    ↓ 异常
                重试 3 次（1s/2s/4s）
                    ↓ 仍失败
                转发到 DLQ + ACK 原消息
```

### 与 v5.0 的对比

| 方面 | v5.0 | v6.0 |
|------|------|------|
| 消费失败 | 消息丢失 | 重试 + DLQ |
| 过期消息 | 仍被消费 | 自动跳过 |
| 重试策略 | 无 | 指数退避 1s/2s/4s |
| 死信处理 | 无 | 转入 {topic}.DLQ |

### 这个版本的局限

| 问题 | 现象 | 下一版解决 |
|------|------|-----------|
| 轮询固定 100ms | 空轮询浪费 CPU | v7.0 自适应退避 |
| 无接口鉴权 | 任何人都能调用 | v7.0 Token 认证 |
| 无可观测性 | 看不到吞吐量和积压 | v7.0 指标暴露 |
| 无延迟消息 | 不支持定时投递 | v7.0 延迟队列 |

### 设计要点

- **重试在客户端**：服务端无状态，重试逻辑在 `MABroker` 里，避免服务端复杂度
- **DLQ 是普通 Topic**：死信队列本质就是另一个 Topic，可以用同样的方式订阅和消费
- **TTL 检查在服务端**：避免客户端拿到过期消息后还要判断，服务端直接跳过更高效
- **过期消息自动 ACK**：跳过时立即推进 offset，避免重复检查同一条过期消息
