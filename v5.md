# v5.0 — 消费者组

## 1. 版本目标

支持同一个 Group 内多个消费者协作消费，每条消息只被组内一个消费者处理，实现水平扩展消费能力。

无 gid 时行为与 v4.0 完全兼容，每个 cid 独立消费全量消息。

**只改服务端**：客户端代码不变，新增 `gid` 参数透传即可。

---

## 2. 核心功能说明

### 独立消费 vs 消费者组

```
独立消费（无 gid，v1.0~v4.0 行为）：

Topic: [msg0, msg1, msg2, msg3]

c1 订阅 → c1 看到全部 4 条，从 offset=0 开始
c2 订阅 → c2 也看到全部 4 条，从 offset=0 开始
→ 同一条消息被 c1 和 c2 各消费一次

消费者组（有 gid）：

c1 sub(gid=g1) ┐
c2 sub(gid=g1) ┘  → g1 组共享一个 offset，从 0 开始

c1 recv(gid=g1) → 拿到 msg0，g1.offset 推进到 1
c2 recv(gid=g1) → 拿到 msg1，g1.offset 推进到 2
c1 recv(gid=g1) → 拿到 msg2，g1.offset 推进到 3
→ 4 条消息被 g1 组合力消费，每条只消费一次
```

### Group offset 的并发安全

多个消费者同时 recv 时，必须保证 offset 的读取和推进是原子的，否则两个消费者可能拿到同一条消息。使用 `synchronized` 保证。

### 接口变化

`sub`、`recv`、`ack`、`stat` 均新增可选参数 `gid`：
- 有 `gid`：走 Group 逻辑，共享 offset
- 无 `gid`：走原有独立逻辑，向后兼容

### 核心数据结构

```
MessageQueue
    ├── subscriptions: Map<cid, Subscription>          ← 独立消费者，不变
    └── groupSubscriptions: Map<gid, Subscription>     ← 新增：组级别 offset
          gid → Subscription{ topic, gid, offset }
          （组内所有成员共享这一个 Subscription）
```

## 4. 代码实现说明

### Subscription.java（增加 groupId）

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class Subscription {
    private String topic;
    private String consumerId;   // 独立消费时是 cid，组消费时是 gid
    private int offset;
    private String groupId;      // 新增，null 表示独立消费
}
```

### MessageQueue.java（核心改动）

```java
public class MessageQueue {
    private final String topic;
    private final Store store;
    private final Map<String, Subscription> subscriptions = new ConcurrentHashMap<>();
    // 组级别 offset：gid → Subscription
    private final Map<String, Subscription> groupSubscriptions = new ConcurrentHashMap<>();

    // 订阅（有 gid 时加入组）
    public void subscribe(String consumerId, String groupId) {
        if (groupId == null) {
            // 独立消费，与 v4.0 一致
            subscriptions.putIfAbsent(consumerId,
                new Subscription(topic, consumerId, 0, null));
        } else {
            // 加入组：个人订阅记录组归属，组级别 offset 只初始化一次
            subscriptions.putIfAbsent(consumerId,
                new Subscription(topic, consumerId, 0, groupId));
            groupSubscriptions.putIfAbsent(groupId,
                new Subscription(topic, groupId, 0, groupId));
        }
    }

    // recv（有 gid 时从组 offset 取，原子推进）
    public Message recv(String consumerId, String groupId) {
        if (groupId == null) {
            return recvIndependent(consumerId);
        }
        return recvGroup(groupId);
    }

    private Message recvIndependent(String consumerId) {
        Subscription sub = subscriptions.get(consumerId);
        if (sub == null || sub.getOffset() >= store.size()) return null;
        int offset = sub.getOffset();
        Message msg = store.read(offset);
        if (msg != null) msg.setOffset(offset);
        return msg;
    }

    // 组消费：读取并原子推进 group offset
    private synchronized Message recvGroup(String groupId) {
        Subscription groupSub = groupSubscriptions.get(groupId);
        if (groupSub == null || groupSub.getOffset() >= store.size()) return null;
        int offset = groupSub.getOffset();
        groupSub.setOffset(offset + 1);   // 立即推进，防止其他消费者拿到同一条
        Message msg = store.read(offset);
        if (msg != null) msg.setOffset(offset);
        return msg;
    }

    // ack（组消费时 ack 不再推进 offset，因为 recv 时已推进）
    public int ack(String consumerId, int offset, String groupId) {
        if (groupId == null) {
            Subscription sub = subscriptions.get(consumerId);
            if (sub != null) {
                sub.setOffset(offset + 1);
                persistSubscriptions();
            }
        }
        // 组消费的 offset 在 recv 时已推进，ack 只做确认，无需再推进
        return offset;
    }

    // stat（有 gid 时返回组的消费进度）
    public Stat stat(String consumerId, String groupId) {
        if (groupId != null) {
            Subscription groupSub = groupSubscriptions.get(groupId);
            int position = groupSub == null ? 0 : groupSub.getOffset();
            return new Stat(store.size(), position);
        }
        Subscription sub = subscriptions.get(consumerId);
        int position = sub == null ? 0 : sub.getOffset();
        return new Stat(store.size(), position);
    }

    // subscribe(consumerId) 保留，向后兼容
    public void subscribe(String consumerId) {
        subscribe(consumerId, null);
    }

    // 其余方法与 v4.0 一致，略
}
```

### MQServer.java（新增 gid 参数）

```java
@GetMapping("/sub")
public Result<String> sub(@RequestParam String t,
                          @RequestParam String cid,
                          @RequestParam(required = false) String gid) {
    getOrCreate(t).subscribe(cid, gid);
    return Result.ok("OK");
}

@GetMapping("/recv")
public Result<Message> recv(@RequestParam String t,
                            @RequestParam String cid,
                            @RequestParam(required = false) String gid) {
    Message msg = getOrCreate(t).recv(cid, gid);
    return Result.ok(msg);
}

@GetMapping("/ack")
public Result<Integer> ack(@RequestParam String t,
                           @RequestParam String cid,
                           @RequestParam int offset,
                           @RequestParam(required = false) String gid) {
    int result = getOrCreate(t).ack(cid, offset, gid);
    return Result.ok(result);
}

@GetMapping("/stat")
public Result<Stat> stat(@RequestParam String t,
                         @RequestParam String cid,
                         @RequestParam(required = false) String gid) {
    Stat stat = getOrCreate(t).stat(cid, gid);
    return Result.ok(stat);
}
```

## 5. 测试流程

启动服务：

```bash
mvn spring-boot:run
```

### 场景一：消费者组基础验证

```bash
# Step 1: 发 6 条消息
for i in $(seq 1 6); do
  curl -s -X POST "http://localhost:8765/mamq/send?t=test.v5" \
    -H "Content-Type: application/json" \
    -d "{\"id\":$i,\"body\":\"msg-$i\",\"headers\":null,\"offset\":null}"
  echo ""
done
# 期望: offset 依次返回 0~5

# Step 2: c1 和 c2 加入同一个 Group g1
curl -s "http://localhost:8765/mamq/sub?t=test.v5&cid=c1&gid=g1"
curl -s "http://localhost:8765/mamq/sub?t=test.v5&cid=c2&gid=g1"
# 期望: 均返回 {"code":1,"data":"OK"}

# Step 3: c1 和 c2 交替消费，验证不重复
echo "--- c1 recv ---"
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c1&gid=g1"
echo ""
echo "--- c2 recv ---"
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c2&gid=g1"
echo ""
echo "--- c1 recv ---"
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c1&gid=g1"
echo ""
# 期望: 三次拿到的是 msg-1、msg-2、msg-3（offset 依次递增，不重复）
```

### 场景二：独立消费者不受 Group 影响

```bash
# solo 不带 gid，独立订阅
curl -s "http://localhost:8765/mamq/sub?t=test.v5&cid=solo"

# solo 从头消费，拿到 msg-1（不受 g1 进度影响）
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=solo"
# 期望: {"code":1,"data":{"id":1,"body":"msg-1",...,"offset":0}}

# g1 继续消费，从 msg-4 开始（不受 solo 影响）
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c1&gid=g1"
# 期望: {"code":1,"data":{"id":4,"body":"msg-4",...,"offset":3}}
```

### 场景三：Group stat 查询

```bash
# 查看 g1 整体消费进度
curl -s "http://localhost:8765/mamq/stat?t=test.v5&cid=c1&gid=g1"
# 期望: {"code":1,"data":{"total":6,"position":4}}
# position=4 表示 g1 已消费到 offset=3，下次从 4 开始

# 查看 solo 独立进度
curl -s "http://localhost:8765/mamq/stat?t=test.v5&cid=solo"
# 期望: {"code":1,"data":{"total":6,"position":1}}
```

### 场景四：消费完所有消息

```bash
# g1 消费剩余 2 条
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c2&gid=g1"
echo ""
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c1&gid=g1"
echo ""

# 再 recv，应返回 null（无更多消息）
curl -s "http://localhost:8765/mamq/recv?t=test.v5&cid=c1&gid=g1"
# 期望: {"code":1,"data":null}
```

## 6. 版本总结

### 学到了什么

v5.0 引入了消费者组，核心思路是：**把 offset 从个人级别提升到组级别**。

```
独立消费：每个 cid 一个 Subscription{offset}，互不影响
组消费：  每个 gid 一个 Subscription{offset}，组内所有成员共享
```

关键设计点：
- `recvGroup` 必须 `synchronized`：读取 offset 和推进 offset 必须原子，否则并发时两个消费者会拿到同一条消息
- 组消费的 `recv` 在取消息时就推进 offset（预占），而不是等 ack 再推进，简化了并发控制
- `ack` 在组模式下只是确认，不再推进 offset（已在 recv 时推进）

### 与 v4.0 的对比

| 方面 | v4.0 | v5.0 |
|------|------|------|
| 消费模式 | 每个 cid 独立全量消费 | 支持组内分摊消费 |
| offset 粒度 | cid 级别 | cid 级别（独立）+ gid 级别（组） |
| 并发安全 | send 加锁 | 新增 recvGroup 加锁 |
| 接口兼容 | — | 无 gid 时行为完全不变 |

### 这个版本的局限

| 问题 | 现象 | 下一版解决 |
|------|------|-----------|
| 消费失败消息丢失 | 回调异常后消息不重试 | v6.0 重试 + DLQ |
| 组 offset 不持久化 | 重启后组消费进度归零 | 可扩展 .sub 格式支持 gid |
| 无消息过期 | 过期消息仍被消费 | v6.0 TTL |
