# v8.0 — 多节点集群

## 1. 版本目标

多个 MAMQ 实例组成集群，Topic 数据按 partition 分片分布在不同节点，实现高可用和水平扩展。

**核心能力**：
- 单节点故障不影响整体服务
- Topic 数据分片，突破单机存储和吞吐瓶颈
- 副本机制保证数据不丢失
- 客户端自动路由到正确节点

## 2. 核心功能说明

### 集群架构

```
                    ZooKeeper / Raft
                    （元数据 + 选举）
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
    Node-1             Node-2             Node-3
  ┌─────────┐        ┌─────────┐        ┌─────────┐
  │ topic-A │        │ topic-A │        │ topic-B │
  │ p0(主)  │        │ p0(副)  │        │ p0(主)  │
  │ topic-B │        │ topic-B │        │ topic-A │
  │ p1(副)  │        │ p1(主)  │        │ p1(副)  │
  └─────────┘        └─────────┘        └─────────┘

客户端 MABroker
      │
      ▼
  路由表（从 ZK 获取）
  topic-A.p0 → Node-1 (主), Node-2 (副)
  topic-A.p1 → Node-2 (主), Node-3 (副)
  topic-B.p0 → Node-3 (主), Node-1 (副)
  topic-B.p1 → Node-2 (主), Node-1 (副)
```

### 核心概念

**Partition（分区）**：Topic 的物理分片，每个 partition 是一个独立的 MessageQueue，有自己的 Store 和 offset。

**Replica（副本）**：每个 partition 有 N 个副本（默认 2），一个 Leader（主）负责读写，其余 Follower（从）同步数据。

**Coordinator（协调者）**：ZooKeeper 或 Raft 集群，存储元数据（Topic → Partition → Node 映射）和 Leader 选举。

**Rebalance（重平衡）**：节点上下线时，自动重新分配 partition 和选举新 Leader。

### 分区策略

```
发送消息时，根据 key 计算 partition：
  partition = hash(key) % partitionCount

无 key 时轮询：
  partition = (lastPartition + 1) % partitionCount

示例：
  topic-A 有 4 个 partition (p0, p1, p2, p3)
  send(key="user-123") → hash("user-123") % 4 = 2 → 路由到 p2
  send(key="user-456") → hash("user-456") % 4 = 1 → 路由到 p1
  send(key=null)       → 轮询 → p0, p1, p2, p3, p0, ...
```

### 副本同步

```
写入流程（同步复制）

Producer → Leader
              │
              ▼
          写入本地 Store
              │
              ▼
          同步到所有 Follower（并行）
              │
              ▼
          所有 Follower ACK 后返回成功

读取流程

Consumer → Leader（只从 Leader 读）
              │
              ▼
          返回消息

Leader 故障

ZooKeeper 检测到 Leader 心跳超时
              │
              ▼
          从 ISR（同步副本集）中选举新 Leader
              │
              ▼
          更新元数据，客户端刷新路由表
```

### ClusterMetadata.java

```java
@Data
public class ClusterMetadata {
    // topic → List<Partition>
    private Map<String, List<Partition>> topicPartitions = new ConcurrentHashMap<>();

    // nodeId → Node{host, port}
    private Map<String, Node> nodes = new ConcurrentHashMap<>();

    public Partition getPartition(String topic, int partitionId) {
        List<Partition> partitions = topicPartitions.get(topic);
        if (partitions == null || partitionId >= partitions.size()) return null;
        return partitions.get(partitionId);
    }
}
```

### ZKCoordinator.java（简化示例）

```java
@Component
public class ZKCoordinator implements Coordinator {

    private CuratorFramework zkClient;
    private String nodeId;

    @PostConstruct
    public void init() {
        zkClient = CuratorFrameworkFactory.newClient(
            "localhost:2181", new ExponentialBackoffRetry(1000, 3));
        zkClient.start();
        nodeId = UUID.randomUUID().toString();
        registerNode();
    }

    private void registerNode() {
        String path = "/mamq/nodes/" + nodeId;
        try {
            zkClient.create()
                .creatingParentsIfNeeded()
                .withMode(CreateMode.EPHEMERAL)  // 临时节点，断线自动删除
                .forPath(path, getNodeInfo().getBytes());
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    @Override
    public ClusterMetadata getMetadata() {
        // 从 ZK 读取 /mamq/metadata 节点
        try {
            byte[] data = zkClient.getData().forPath("/mamq/metadata");
            return JSON.parseObject(new String(data), ClusterMetadata.class);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    @Override
    public void electLeader(String topic, int partitionId) {
        String path = "/mamq/leaders/" + topic + "/" + partitionId;
        try {
            // 使用 ZK 的临时顺序节点实现 Leader 选举
            zkClient.create()
                .creatingParentsIfNeeded()
                .withMode(CreateMode.EPHEMERAL_SEQUENTIAL)
                .forPath(path + "/node-", nodeId.getBytes());

            // 序号最小的节点成为 Leader
            List<String> children = zkClient.getChildren().forPath(path);
            Collections.sort(children);
            String leader = new String(zkClient.getData()
                .forPath(path + "/" + children.get(0)));

            if (leader.equals(nodeId)) {
                System.out.println("我是 Leader: " + topic + ".p" + partitionId);
            }
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
```

### ReplicaManager.java

```java
@Component
public class ReplicaManager {

    @Autowired
    private ZKCoordinator coordinator;

    // 同步写入所有副本
    public boolean replicateToFollowers(String topic, int partitionId, Message msg) {
        ClusterMetadata metadata = coordinator.getMetadata();
        Partition partition = metadata.getPartition(topic, partitionId);

        List<String> followers = partition.getReplicaIds().stream()
            .filter(id -> !id.equals(partition.getLeaderId()))
            .collect(Collectors.toList());

        // 并行同步到所有 Follower
        CountDownLatch latch = new CountDownLatch(followers.size());
        AtomicInteger successCount = new AtomicInteger(0);

        for (String followerId : followers) {
            CompletableFuture.runAsync(() -> {
                try {
                    Node node = metadata.getNodes().get(followerId);
                    String url = "http://" + node.getHost() + ":" + node.getPort()
                        + "/mamq/replica/append?t=" + topic + "&p=" + partitionId;
                    String response = HttpUtil.post(url, JSON.toJSONString(msg));
                    if (response.contains("\"code\":1")) {
                        successCount.incrementAndGet();
                    }
                } finally {
                    latch.countDown();
                }
            });
        }

        try {
            latch.await(5, TimeUnit.SECONDS);
        } catch (InterruptedException e) {
            return false;
        }

        // 至少一半副本成功才算成功（简化的 Quorum）
        return successCount.get() >= followers.size() / 2;
    }
}
```

### PartitionRouter.java

```java
@Component
public class PartitionRouter {

    private final AtomicInteger roundRobinCounter = new AtomicInteger(0);

    public int selectPartition(String topic, String key, int partitionCount) {
        if (key == null || key.isEmpty()) {
            // 无 key，轮询
            return roundRobinCounter.getAndIncrement() % partitionCount;
        }
        // 有 key，hash
        return Math.abs(key.hashCode()) % partitionCount;
    }
}
```

### MAProducer.java（增加分区路由）

```java
public class MAProducer {

    private final String baseUrl;
    private ClusterMetadata metadata;
    private PartitionRouter router = new PartitionRouter();

    public int send(String topic, String key, Message<?> message) {
        // 从 ZK 获取最新元数据
        refreshMetadata();

        List<Partition> partitions = metadata.getTopicPartitions().get(topic);
        int partitionId = router.selectPartition(topic, key, partitions.size());
        Partition partition = partitions.get(partitionId);

        // 路由到 Leader 节点
        Node leader = metadata.getNodes().get(partition.getLeaderId());
        String url = "http://" + leader.getHost() + ":" + leader.getPort()
            + "/mamq/send?t=" + topic + "&p=" + partitionId;

        String body = JSON.toJSONString(message);
        String response = HttpUtil.post(url, body);
        Result<Integer> result = JSON.parseObject(response,
            new TypeReference<Result<Integer>>() {});
        return result.getData();
    }

    private void refreshMetadata() {
        // 从 ZK 获取元数据（实际应加缓存，定期刷新）
        String zkUrl = "http://localhost:2181/mamq/metadata";

}
```

### MQServer.java（新增副本同步接口）

```java
@RestController
@RequestMapping("/mamq")
public class MQServer {

    @Autowired
    private ReplicaManager replicaManager;

    @Autowired
    private ZKCoordinator coordinator;

    // 原有 send 接口改为支持 partition
    @PostMapping("/send")
    public Result<Integer> send(@RequestParam String t,
                                @RequestParam(defaultValue = "0") int p,
                                @RequestBody Message message) {
        // 检查当前节点是否是该 partition 的 Leader
        ClusterMetadata metadata = coordinator.getMetadata();
        Partition partition = metadata.getPartition(t, p);
        if (!partition.getLeaderId().equals(coordinator.getNodeId())) {
            return Result.error("Not leader for " + t + ".p" + p);
        }

        // 写入本地
        int offset = getOrCreate(t, p).send(message);

        // 同步到 Follower
        boolean replicated = replicaManager.replicateToFollowers(t, p, message);
        if (!replicated) {
            return Result.error("Replication failed");
        }

        return Result.ok(offset);
    }

    // 新增：Follower 接收 Leader 同步的消息
    @PostMapping("/replica/append")
    public Result<String> replicaAppend(@RequestParam String t,
                                        @RequestParam int p,
                                        @RequestBody Message message) {
        getOrCreate(t, p).send(message);
        return Result.ok("OK");
    }
}
```

### 场景一：分区写入验证

```bash
# 创建 Topic，4 个 partition，2 个副本

  -H "Content-Type: application/json" \
  -d '{"topic":"test.v8","partitions":4,"replicas":2}'

# 发送 10 条消息，观察分区分布
for i in $(seq 1 10); do
  curl -s -X POST "http://localhost:8765/mamq/send?t=test.v8" \
    -H "Content-Type: application/json" \
    -d "{\"id\":$i,\"body\":\"msg-$i\",\"headers\":null,\"offset\":null}"

done

# 查看各 partition 的消息数
for p in 0 1 2 3; do
  echo "=== Partition $p ==="
  curl -s "http://localhost:8765/mamq/stat?t=test.v8&p=$p&cid=c1"
  echo ""
done
# 期望: 消息均匀分布在 4 个 partition
```

## 6. 版本总结

### 学到了什么

v8.0 实现了分布式消息队列的核心架构：

**分区（Partition）**：Topic 的物理分片，突破单机瓶颈，每个 partition 独立存储和消费。

**副本（Replica）**：数据冗余，Leader 故障时自动切换，保证高可用。

**协调者（Coordinator）**：ZooKeeper 管理元数据和选举，是集群的"大脑"。

**客户端路由**：根据 key 计算 partition，自动找到 Leader 节点，对业务透明。

```
集群架构

ZooKeeper（元数据 + 选举）
      │
      ├─► Node-1: topic-A.p0(主), topic-B.p1(副)
      ├─► Node-2: topic-A.p1(主), topic-A.p0(副)
      └─► Node-3: topic-B.p0(主), topic-A.p1(副)

写入流程：Producer → Leader → 同步到 Follower → 返回成功
故障切换：Leader 挂掉 → ZK 选举新 Leader → 客户端刷新路由
```

### 与 v7.0 的对比

| 方面 | v7.0 | v8.0 |
|------|------|------|
| 部署模式 | 单机 | 多节点集群 |
| 可用性 | 单点故障 | Leader 自动切换 |
| 存储容量 | 单机磁盘上限 | 分区分布，水平扩展 |
| 吞吐量 | 单机瓶颈 | 多节点并行写入 |
| 数据安全 | 单机丢失即丢失 | 副本冗余 |

### 这个版本的局限

| 问题 | 现象 | 未来解决 |
|------|------|----------|
| 同步复制延迟高 | 写入等待所有副本 ACK | 异步复制 + ISR 机制 |
| 无消息过滤 | 消费者拿到所有消息 | v9.0 过滤规则 |
| 无管理界面 | 只能 API 查看集群状态 | v10.0 Web UI |
| Rebalance 停服 | 节点上下线时短暂不可用 | 优化 Rebalance 算法 |

### 设计要点

- **分区 key 一致性**：同一 key 的消息总是路由到同一 partition，保证顺序
- **ISR（In-Sync Replicas）**：只有同步副本才能参与选举，保证数据不丢
- **Quorum 机制**：至少一半副本成功才算成功，平衡可用性和一致性
- **临时节点心跳**：ZK 的临时节点自动检测故障，无需额外心跳线程
- **客户端缓存元数据**：避免每次请求都查 ZK，定期刷新即可

### 集群运维要点

- **节点数建议**：至少 3 个节点，副本数 2-3
- **ZK 部署**：生产环境 ZK 也要集群部署（3/5/7 节点）
- **监控指标**：新增 `mamq_partition_lag`、`mamq_replica_lag`、`mamq_leader_election_total`
- **扩容流程**：新增节点 → ZK 自动发现 → Rebalance 迁移部分 partition
- **缩容流程**：停止节点 → ZK 检测下线 → Rebalance 迁移 partition 到其他节点

v8.0 已具备分布式消息队列的核心能力，可支撑大规模生产环境。
