# v1.0 — 内存队列，跑通链路

## 1. 版本目标

用最少的代码跑通 `send → sub → recv → ack` 完整链路，所有数据存内存，重启即丢失。

本版本的核心价值不是功能完整，而是**看清结构**：一个消息系统由哪些角色组成、每个角色做什么、它们之间如何协作。

**刻意不做**：持久化、客户端封装、batch 接口。复杂度为零，结构一目了然。

---

## 2. 核心功能说明

### 4 个角色

| 角色 | 类比 | 职责 |
|------|------|------|
| Producer | 往传送带放零件的工人 | 向 Topic 写入消息 |
| Consumer | 从传送带取零件的工人 | 从 Topic 读取消息 |
| Topic | 一条传送带 | 消息的逻辑分类，每个 Topic 独立存储 |
| Broker（MQServer） | 工厂管理员 | 接收 HTTP 请求，路由到对应 Topic |

### 4 个动作

| 动作 | HTTP 接口 | 说明 |
|------|-----------|------|
| send | `POST /mamq/send?t={topic}` | Producer 写入一条消息，返回该消息的 offset（下标） |
| sub | `GET /mamq/sub?t={topic}&cid={consumerId}` | Consumer 订阅 Topic，初始化 offset=0 的书签 |
| recv | `GET /mamq/recv?t={topic}&cid={consumerId}` | Consumer 读取当前 offset 的消息，不推进 offset |
| ack | `GET /mamq/ack?t={topic}&cid={consumerId}&offset={n}` | Consumer 确认消费，offset 推进到 n+1 |

### Offset 语义

```
消息列表:  [msg0,  msg1,  msg2,  msg3,  msg4]
下标:        0      1      2      3      4

Subscription{cid="c1", offset=2}
  → 表示 c1 已消费到 offset=1，下次 recv 从 offset=2 开始取
  → ack(offset=2) 后，offset 推进到 3
```

offset 就是消息在 ArrayList 中的下标，简单直接。

### stat 接口

`GET /mamq/stat?t={topic}&cid={consumerId}` 返回：
- `total`：该 Topic 共有多少条消息
- `position`：该消费者当前消费到第几条（即当前 offset）

## 3. 代码结构

```
src/main/java/cn/malinghan/mamq/
├── model/
│   ├── Message.java          # 消息体：id + body + headers
│   ├── Subscription.java     # 订阅关系：topic + consumerId + offset
│   ├── Stat.java             # 消费统计：total + position
│   └── Result.java           # HTTP 响应包装：code + data
├── server/
│   ├── MessageQueue.java     # 核心：Topic 的消息存储 + 订阅管理
│   └── MQServer.java         # HTTP 接入层：REST Controller
└── MamqApplication.java      # Spring Boot 启动类
```

### 各文件职责

```
MQServer（HTTP层）
    │  接收请求，解析参数
    │  路由到对应 MessageQueue
    ▼
MessageQueue（核心层，每个 Topic 一个实例）
    ├── messages: ArrayList<Message>          ← 消息存储
    ├── subscriptions: ConcurrentHashMap      ← 订阅关系
    │     cid → Subscription{offset}
    ├── send(msg)    → messages.add(msg)，返回 offset
    ├── recv(cid)    → messages.get(sub.offset)
    ├── ack(cid, n)  → sub.offset = n + 1
    └── stat(cid)    → {total, position}

MQServer 维护：
    queues: ConcurrentHashMap<String, MessageQueue>
      topic → MessageQueue 实例
    Topic 不存在时自动创建（动态 Topic）
```

### Result.java

```java
@Data
@AllArgsConstructor
public class Result<T> {
    private int code;    // 1=成功, 0=失败
    private T data;

    public static <T> Result<T> ok(T data) {
        return new Result<>(1, data);
    }
    public static <T> Result<T> error(String msg) {
        return new Result<>(0, (T) msg);
    }
}
```

### MessageQueue.java

```java
public class MessageQueue {
    private final String topic;
    // 消息存储：下标即 offset
    private final List<Message> messages = new ArrayList<>();
    // 订阅关系：consumerId → Subscription
    private final Map<String, Subscription> subscriptions = new ConcurrentHashMap<>();

    public MessageQueue(String topic) {
        this.topic = topic;
    }

    // 写入消息，返回 offset（即当前 size - 1）
    public synchronized int send(Message message) {
        messages.add(message);
        return messages.size() - 1;
    }

    // 订阅：初始化 offset=0 的书签
    public void subscribe(String consumerId) {
        subscriptions.putIfAbsent(consumerId,
            new Subscription(topic, consumerId, 0));
    }

    // 取消订阅
    public void unsubscribe(String consumerId) {
        subscriptions.remove(consumerId);
    }

    // 读取当前 offset 的消息，不推进 offset
    public Message recv(String consumerId) {
        Subscription sub = subscriptions.get(consumerId);
        if (sub == null || sub.getOffset() >= messages.size()) {
            return null;
        }
        return messages.get(sub.getOffset());
    }

    // 确认消费，offset 推进到 offset+1
    public int ack(String consumerId, int offset) {
        Subscription sub = subscriptions.get(consumerId);
        if (sub != null) {
            sub.setOffset(offset + 1);
        }
        return offset;
    }

    // 统计
    public Stat stat(String consumerId) {
        Subscription sub = subscriptions.get(consumerId);
        int position = sub == null ? 0 : sub.getOffset();
        return new Stat(messages.size(), position);
    }
}
```

### MQServer.java

```java
@RestController
@RequestMapping("/mamq")
public class MQServer {

    // Topic → MessageQueue，动态创建
    private final Map<String, MessageQueue> queues = new ConcurrentHashMap<>();

    private MessageQueue getOrCreate(String topic) {
        return queues.computeIfAbsent(topic, MessageQueue::new);
    }

    @PostMapping("/send")
    public Result<Integer> send(@RequestParam String t,
                                @RequestBody Message message) {
        int offset = getOrCreate(t).send(message);
        return Result.ok(offset);
    }

    @GetMapping("/sub")
    public Result<String> sub(@RequestParam String t,
                              @RequestParam String cid) {
        getOrCreate(t).subscribe(cid);
        return Result.ok("OK");
    }

    @GetMapping("/unsub")
    public Result<String> unsub(@RequestParam String t,
                                @RequestParam String cid) {
        getOrCreate(t).unsubscribe(cid);
        return Result.ok("OK");
    }

    @GetMapping("/recv")
    public Result<Message> recv(@RequestParam String t,
                                @RequestParam String cid) {
        Message msg = getOrCreate(t).recv(cid);
        return Result.ok(msg);
    }

    @GetMapping("/ack")
    public Result<Integer> ack(@RequestParam String t,
                               @RequestParam String cid,
                               @RequestParam int offset) {
        int result = getOrCreate(t).ack(cid, offset);
        return Result.ok(result);
    }

    @GetMapping("/stat")
    public Result<Stat> stat(@RequestParam String t,
                             @RequestParam String cid) {
        Stat stat = getOrCreate(t).stat(cid);
        return Result.ok(stat);
    }
}
```

## 5. 测试流程

启动服务：

```bash
mvn spring-boot:run
```

### 场景一：基础链路验证

```bash
# Step 1: 发送 3 条消息到 Topic "test.t1"
curl -s -X POST "http://localhost:8765/mamq/send?t=test.t1" \
  -H "Content-Type: application/json" \
  -d '{"id":1,"body":"hello","headers":{}}'
# 期望: {"code":1,"data":0}   ← offset=0

curl -s -X POST "http://localhost:8765/mamq/send?t=test.t1" \
  -H "Content-Type: application/json" \
  -d '{"id":2,"body":"world","headers":{}}'
# 期望: {"code":1,"data":1}   ← offset=1

curl -s -X POST "http://localhost:8765/mamq/send?t=test.t1" \
  -H "Content-Type: application/json" \
  -d '{"id":3,"body":"mamq","headers":{}}'
# 期望: {"code":1,"data":2}   ← offset=2

# Step 2: 消费者 c1 订阅
curl -s "http://localhost:8765/mamq/sub?t=test.t1&cid=c1"
# 期望: {"code":1,"data":"OK"}

# Step 3: recv 第 1 条（offset=0）
curl -s "http://localhost:8765/mamq/recv?t=test.t1&cid=c1"
# 期望: {"code":1,"data":{"id":1,"body":"hello","headers":{}}}

# Step 4: 再次 recv，不 ack 的话还是拿到同一条
curl -s "http://localhost:8765/mamq/recv?t=test.t1&cid=c1"
# 期望: {"code":1,"data":{"id":1,"body":"hello","headers":{}}}  ← 同一条，offset 未推进

# Step 5: ACK offset=0，推进到 1
curl -s "http://localhost:8765/mamq/ack?t=test.t1&cid=c1&offset=0"
# 期望: {"code":1,"data":0}

# Step 6: 再 recv，拿到第 2 条
curl -s "http://localhost:8765/mamq/recv?t=test.t1&cid=c1"
# 期望: {"code":1,"data":{"id":2,"body":"world","headers":{}}}

# Step 7: ACK offset=1，再 recv 拿第 3 条
curl -s "http://localhost:8765/mamq/ack?t=test.t1&cid=c1&offset=1"
curl -s "http://localhost:8765/mamq/recv?t=test.t1&cid=c1"
# 期望: {"code":1,"data":{"id":3,"body":"mamq","headers":{}}}

# Step 8: ACK 最后一条，再 recv 返回 null（无更多消息）
curl -s "http://localhost:8765/mamq/ack?t=test.t1&cid=c1&offset=2"
curl -s "http://localhost:8765/mamq/recv?t=test.t1&cid=c1"
# 期望: {"code":1,"data":null}
```

### 场景二：多消费者独立消费

```bash
# 同一 Topic，两个消费者各自独立维护 offset
curl -s "http://localhost:8765/mamq/sub?t=test.t1&cid=c2"

# c2 从头开始消费，不受 c1 影响
curl -s "http://localhost:8765/mamq/recv?t=test.t1&cid=c2"
# 期望: {"code":1,"data":{"id":1,"body":"hello",...}}  ← 从第 1 条开始
```

### 场景三：stat 统计

```bash
# c1 已消费完 3 条（offset=3），c2 还在 offset=0
curl -s "http://localhost:8765/mamq/stat?t=test.t1&cid=c1"
# 期望: {"code":1,"data":{"total":3,"position":3}}

curl -s "http://localhost:8765/mamq/stat?t=test.t1&cid=c2"
# 期望: {"code":1,"data":{"total":3,"position":0}}
```

### 场景四：动态 Topic 创建

```bash
# 向一个从未存在的 Topic 发消息，自动创建
curl -s -X POST "http://localhost:8765/mamq/send?t=brand.new.topic" \
  -H "Content-Type: application/json" \
  -d '{"id":1,"body":"auto-created","headers":{}}'
# 期望: {"code":1,"data":0}  ← Topic 自动创建，消息写入成功
```

### 场景五：重启验证（内存特性）

```bash
# 发消息后重启服务（Ctrl+C 后重新 mvn spring-boot:run）
# 重启后查询
curl -s "http://localhost:8765/mamq/stat?t=test.t1&cid=c1"
# 期望: {"code":1,"data":{"total":0,"position":0}}  ← 数据全部丢失，符合预期
```

## 6. 版本总结

### 学到了什么

通过 v1.0，可以清楚地看到一个消息系统的最小骨架：

```
Producer → [send] → Topic(ArrayList) → [recv] → Consumer
                                              ↑
                                           [ack] 推进 offset
```

- **Topic** 就是一个带下标的列表，消息按写入顺序排列
- **Subscription** 就是一个书签，记录每个消费者读到哪里了
- **ACK** 就是把书签往后翻一页
- **MQServer** 就是一个路由层，把 HTTP 请求转发给对应的 Topic

### 这个版本的局限

| 问题 | 现象 | 下一版解决 |
|------|------|-----------|
| 重启数据丢失 | 服务重启后消息全没了 | v2.0 持久化存储 |
| 内存无上限 | 消息越积越多，最终 OOM | v2.0 文件存储 |
| 无客户端封装 | 调用方要手写 HTTP 请求 | v3.0 客户端封装 |
| 无 push 模式 | 消费者只能主动轮询 | v3.0 MABroker |

### 核心设计决策

- `send` 返回 offset 而不是消息 ID：offset 是位置坐标，消费者用它来 ack
- `recv` 不自动推进 offset：必须显式 ack，实现"至少一次投递"语义
- `ConcurrentHashMap` 管理订阅：多消费者并发订阅安全
- `synchronized send`：写入加锁，保证 offset 连续不跳号
