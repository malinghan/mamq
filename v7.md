# v7.0 — 生产就绪

## 1. 版本目标

具备基本的生产运行能力，包括自适应轮询、接口鉴权、可观测性指标、延迟消息。

**四个生产级特性**：
- 自适应轮询：无消息时指数退避，最长 5s，减少空轮询
- 接口鉴权：Token 认证，防止未授权访问
- 可观测性：暴露 Prometheus 指标，监控吞吐量和积压
- 延迟消息：支持定时投递，消息在指定时间后才可消费

---

## 2. 核心功能说明

### 自适应轮询

```
v6.0 固定轮询：
  recv() → 无消息 → sleep(100ms) → recv() → 无消息 → sleep(100ms) ...
  问题：空轮询浪费 CPU，100ms 固定间隔不够灵活

v7.0 自适应退避：
  recv() → 无消息 → sleep(100ms)
  recv() → 无消息 → sleep(200ms)
  recv() → 无消息 → sleep(400ms)
  recv() → 无消息 → sleep(800ms)
  recv() → 无消息 → sleep(1600ms)
  recv() → 无消息 → sleep(3200ms)
  recv() → 无消息 → sleep(5000ms)  ← 最长 5s
  recv() → 有消息 → 立即处理 → sleep(100ms) 重置
```

### 接口鉴权

所有 HTTP 接口要求携带 `X-Token` header，值为配置的密钥（如 `mamq-secret`），否则返回 401。

```
application.properties:
  mamq.auth.token=mamq-secret
  mamq.auth.enabled=true

AuthFilter 拦截所有 /mamq/* 请求：
  request.getHeader("X-Token") == mamq.auth.token ? 放行 : 401
```

### 可观测性指标

集成 Spring Boot Actuator + Micrometer，暴露 Prometheus 格式指标：

- `mamq_send_total{topic="T"}` — Topic T 的累计发送消息数
- `mamq_recv_total{topic="T"}` — Topic T 的累计消费消息数
- `mamq_lag{topic="T",consumer="C"}` — 消费者 C 在 Topic T 的积压量（total - position）

访问 `http://localhost:8765/actuator/prometheus` 查看所有指标。

### 延迟消息

发送时携带 `X-delay-ms` header（毫秒），消息不立即写入 Store，而是放入内存延迟队列，后台线程每秒扫描，到期后写入 Store。

```
send(msg with X-delay-ms=5000)
      │
      ▼
  MessageQueue.delayQueue.offer(msg, deliverTime)
      │
      ▼
  后台线程每秒扫描
      │
      ▼
  deliverTime <= now ?
  ┌───┴───┐
  │ Yes   │ No → 继续等待
  ▼
  store.write(msg) → 对消费者可见
```

## 4. 代码实现说明

### MABroker.java（自适应轮询）

```java
public class MABroker {
    private static final long INITIAL_WAIT = 100;
    private static final long MAX_WAIT = 5000;

    @SuppressWarnings("unchecked")
    private <T> void pollOnce(MAConsumer<T> consumer) throws InterruptedException {
        Message<T> msg = consumer.recv();
        if (msg == null) {
            // 无消息，指数退避
            long wait = Math.min(consumer.waitInterval * 2, MAX_WAIT);
            consumer.waitInterval = wait;
            Thread.sleep(wait);
            return;
        }

        // 有消息，重置等待时间
        consumer.waitInterval = INITIAL_WAIT;

        if (consumer.listener != null) {
            boolean success = processWithRetry(consumer, msg);
            if (success) {
                consumer.ack(msg.getOffset());
            } else {
                sendToDLQ(consumer, msg);
                consumer.ack(msg.getOffset());
            }
        }
    }
}
```

### AuthFilter.java

```java
@Component
@Order(1)
public class AuthFilter implements Filter {

    @Value("${mamq.auth.enabled:false}")
    private boolean authEnabled;

    @Value("${mamq.auth.token:}")
    private String authToken;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        if (!authEnabled) {
            chain.doFilter(request, response);
            return;
        }

        HttpServletRequest req = (HttpServletRequest) request;
        HttpServletResponse resp = (HttpServletResponse) response;

        if (!req.getRequestURI().startsWith("/mamq/")) {
            chain.doFilter(request, response);
            return;
        }

        String token = req.getHeader("X-Token");
        if (authToken.equals(token)) {
            chain.doFilter(request, response);
        } else {
            resp.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            resp.getWriter().write("{\"code\":0,\"data\":\"Unauthorized\"}");
        }
    }
}
```

### MQMetrics.java

```java
@Component
public class MQMetrics {

    private final MeterRegistry registry;
    private final Map<String, Counter> sendCounters = new ConcurrentHashMap<>();
    private final Map<String, Counter> recvCounters = new ConcurrentHashMap<>();

    public MQMetrics(MeterRegistry registry) {
        this.registry = registry;
    }

    public void recordSend(String topic) {
        sendCounters.computeIfAbsent(topic, t ->
            Counter.builder("mamq.send.total")
                .tag("topic", t)
                .register(registry)
        ).increment();
    }

    public void recordRecv(String topic) {
        recvCounters.computeIfAbsent(topic, t ->
            Counter.builder("mamq.recv.total")
                .tag("topic", t)
                .register(registry)
        ).increment();
    }

    public void registerLagGauge(String topic, String consumerId, Supplier<Number> lagSupplier) {
        Gauge.builder("mamq.lag", lagSupplier)
            .tag("topic", topic)
            .tag("consumer", consumerId)
            .register(registry);
    }
}
```

### MessageQueue.java（延迟队列 + 指标）

```java
public class MessageQueue {
    private final String topic;
    private final Store store;
    private final Map<String, Subscription> subscriptions = new ConcurrentHashMap<>();
    private final Map<String, Subscription> groupSubscriptions = new ConcurrentHashMap<>();
    private final File subFile;

    // 延迟队列：按投递时间排序
    private final PriorityQueue<DelayedMessage> delayQueue = new PriorityQueue<>(
        Comparator.comparingLong(DelayedMessage::getDeliverTime)
    );

    @Autowired
    private MQMetrics metrics;

    @Data
    @AllArgsConstructor
    static class DelayedMessage {
        private Message message;
        private long deliverTime;  // 投递时间戳（毫秒）
    }

    public int send(Message message) {
        // 检查是否延迟消息
        if (message.getHeaders() != null) {
            String delayMs = message.getHeaders().get("X-delay-ms");
            if (delayMs != null) {
                long delay = Long.parseLong(delayMs);
                long deliverTime = System.currentTimeMillis() + delay;
                synchronized (delayQueue) {
                    delayQueue.offer(new DelayedMessage(message, deliverTime));
                }
                return -1;  // 延迟消息返回特殊值，表示未立即写入
            }
        }

        int offset = store.write(message);
        metrics.recordSend(topic);
        return offset;
    }

    public Message recv(String consumerId, String groupId) {
        Message msg = (groupId == null) ? recvIndependentWithTTL(consumerId)
                                        : recvGroupWithTTL(groupId);
        if (msg != null) {
            metrics.recordRecv(topic);
        }
        return msg;
    }

    // 后台线程定期调用，将到期消息写入 Store
    public void processDelayedMessages() {
        long now = System.currentTimeMillis();
        synchronized (delayQueue) {
            while (!delayQueue.isEmpty() && delayQueue.peek().getDeliverTime() <= now) {
                DelayedMessage dm = delayQueue.poll();
                store.write(dm.getMessage());
                metrics.recordSend(topic);
            }
        }
    }

    // 其他方法与 v6.0 一致，略
}
```

### MQConfig.java（启动延迟队列扫描线程）

```java
@Configuration
public class MQConfig {

    @Autowired
    private MQServer mqServer;

    @PostConstruct
    public void startDelayedMessageScanner() {
        ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor();
        scheduler.scheduleAtFixedRate(() -> {
            mqServer.getAllQueues().forEach((topic, queue) -> {
                try {
                    queue.processDelayedMessages();
                } catch (Exception e) {
                    System.err.println("处理延迟消息失败: " + topic + ", " + e.getMessage());
                }
            });
        }, 1, 1, TimeUnit.SECONDS);
    }
}
```

## 6. 版本总结

### 学到了什么

v7.0 补齐了生产环境的四个关键能力：

**自适应轮询**：空闲时减少 CPU 消耗，繁忙时保持低延迟，兼顾性能和资源利用率。

**接口鉴权**：防止未授权访问，生产环境必备。

**可观测性**：通过 Prometheus 指标实时监控吞吐量和积压，及时发现问题。

**延迟消息**：支持定时任务场景（如订单超时取消、定时推送）。

```
生产就绪 = 性能优化 + 安全 + 可观测 + 高级特性

自适应轮询：100ms → 200ms → 400ms → ... → 5000ms（指数退避）
鉴权：X-Token header 验证
指标：mamq_send_total / mamq_recv_total / mamq_lag
延迟消息：内存优先级队列 + 后台扫描线程
```

### 与 v6.0 的对比

| 方面 | v6.0 | v7.0 |
|------|------|------|
| 轮询策略 | 固定 100ms | 指数退避 100ms~5s |
| 接口安全 | 无鉴权 | Token 认证 |
| 可观测性 | 无 | Prometheus 指标 |
| 延迟消息 | 不支持 | 支持 X-delay-ms |
| 生产就绪 | 否 | 是 |

### 这个版本的局限

| 问题 | 现象 | 未来解决 |
|------|------|----------|
| 单机部署 | 单点故障 | v8.0 集群 |
| 无消息过滤 | 消费者拿到所有消息 | v9.0 过滤规则 |
| 无管理界面 | 只能 curl 查看 | v10.0 Web UI |
| 延迟队列在内存 | 重启丢失 | 可持久化到文件 |

### 设计要点

- **自适应退避**：指数增长避免过快到达上限，5s 上限平衡响应速度和资源消耗
- **鉴权在 Filter**：统一拦截，Controller 无需改动
- **指标用 Micrometer**：标准库，兼容 Prometheus/Grafana 生态
- **延迟队列用优先级队列**：按投递时间排序，每秒扫描队首即可，O(log n) 插入
- **延迟消息返回 -1**：区分立即写入（返回 offset）和延迟写入（返回 -1）
